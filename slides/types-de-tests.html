Différents niveaux de tests

Tests d'acceptation

Utilisation de l'application en "situation réelle".

  • Niveau de détail utile pour les parties prenantes,
  • Efficaces pour garantir que le système fonctionne du point de vue de l'utilisateur final.
  • Fragiles (facilement cassés par les changements prévus de l'interface utilisateur),
  • Plus grande marge de faux positifs
  • Coûteux à écrire et à entretenir

RAZ des données avant exécution

Tests d'intégration

Testent l'assemblage des composants unitaires entre eux.

  • Rapides à écrire
  • Permettent de tester la configuration d'une l'application
  • Risque d'obtention de tests dépendants
  • Complexes à maintenir (les point d'intégration sont partagés)

Tests unitaires

Tests élémentaires de composants logiciel (ensemble de fonctions, classe ...).

  • Rapides à exécuter
  • Rapides à mettre en place (en théorie)
  • Feedback efficace en cas de problème
  • Porté limitée

Couverture de code

  • Pourcentage de classes, fonctions, lignes de code couverts par les tests
  • Donne un appréciation des méthodes critiques couvertes par les tests
  • Fournit des métriques et permet de suivre l'évolution des tests
  • Ne donne pas d'indication sur la qualité des tests effectués
  • Temps d'exécution : rapide => long
  • Coût de création et maintenance : bas => élevé
  • Précision / fiabilité: forte => faible
  • Mise en place plus tardive : nécessité d'une interface utilisateur, d'une route d'API, d'IO
  • Détection tardive des erreurs
  • Plus liés à la présentation qu'aux données réelles
