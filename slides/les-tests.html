Les tests

Différentes perspectives, différents objectifs

Perspective Objectif
Développeur Détecter les bugs
QA Évaluer la qualité
Testeur Valider le fonctionnel par rapport aux spécifications
Client / PO Vérifier le fonctionnement

Objectifs

  • Découvrir / prévenir les bugs
  • Accélérer les développements
  • Permettre le refactoring
  • Documenter le code
  • Détecter les problèmes de design
  1. Valider le fonctionnement d'une méthode et les effets de bord potentiels
  2. On ne teste pas tout à chaque modification de code
  3. Assurer les modifications sans casse
  4. Option testdox (cf dernière partie)
  5. Code difficilement testable souvent lié à un problème de conception

Pour être utiles
et donc utilisés

  • Rapides
  • Indépendants de l'implémentation
  • Décrire le comportement du système
  • Avoir un résultat prédictible
  • Isolés les uns des autres
  • Simples à mettre en place
  • Pouvoir être exécutés souvent, si besoin par portions
  • Tester les usages possibles, scinder les cas d'utilisation des classes (pas forcément une classe de test pour une classe)
  • Ex : ne pas affecter à la volée des propriétés uniquement initialisables
  • Utiliser des fixtures, ne pas se baser sur des propriétés du système
  • Éviter les erreurs en cascade en cas de problème
  • Créer des assertions pour des assertions groupées vérifiant une règle
